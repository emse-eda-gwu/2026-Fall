Why version control + agents belong together

Version control

  • A save point you can return to
  • A cloud backup of your work
  • A record of what changed and when

Agentic workflow

  • You direct, the agent edits your files
  • Fast — and occasionally wrong
  • You need a safety net


You commit often precisely so it’s safe to let an agent
loose on your files. The two are one workflow.

Part 1 — GitHub Desktop

Three words, that’s the whole model

repo

A project folder that tracks its own history

commit

A labeled save point — “this is a state worth remembering”

push

Send your commits up to GitHub (the cloud copy)


No git in a terminal. Everything today is a button in GitHub Desktop.

Practice 1

Make a repo and push to it.

  1. GitHub Desktop → File › New Repository → name it eda-w2-practice
  2. Publish it to GitHub (make it private)
  3. Add a line to README.md, then Commit (write a message!)
  4. Push — then open it on github.com and confirm it’s there
  5. Change the README again → Commit → Push → look at History

Practice 2

Let Claude import some data — and read what it did.

  1. Drop flights.csv into your repo’s data/ folder
  2. Ask Claude: “Import data/flights.csv and give me a quick summary of what’s in it.”
  3. Read the code it wrote. Not just the summary — the actual code.
  4. Tell a neighbor: what did it do, and how do you know it’s right?

The habit to build now

The summary Claude prints is a claim.

The code it wrote is the evidence.


Always read the evidence, not just the claim.

Part 3 — The whole loop

Ask → verify → commit → push

Now we string it together into one motion:

  1. Ask Claude to make a chart from the data
  2. Verify it — do the numbers behind the chart match reality?
  3. Commit the chart + code with a clear message
  4. Push it to GitHub


This is the exact loop you’ll run all semester. Everything else is details.

Practice 3

Run the full loop yourself.

  1. Ask Claude to make a chart answering a question about flights.csv
  2. Verify: pick one bar/point and check it against the data by hand
  3. Does the chart actually say what you asked? If not, tell Claude what’s wrong
  4. Commit (chart + code) → Push to GitHub

Verify.

Part 4 — Where it goes wrong

The agent will be wrong with confidence

It won’t say “I’m not sure.” It will hand you a clean, plausible,
wrong answer and move on. Common failure modes:

  • Hallucinated column names
  • Silently dropped rows (NAs, bad joins)
  • Wrong join direction
  • A chart that misleads (bad axis, wrong aggregation)
  • A made-up number that “looks right”
  • Code that isn’t reproducible

Let me show you one live.

Your verification checklist

Run these every time — they’re the skill this whole course is really about:

  • Did row / column counts change the way I expected?
  • Did I read the code, not just the prose summary?
  • Did I spot-check a few values against the source file?
  • Were rows silently dropped (NA handling, inner vs. left join)?
  • Do totals / aggregates sanity-check?
  • Is it reproducible — or did it hard-code an answer?

We’ll come back to this list all semester. Screenshot it.

Practice 4

Catch the bug.

I’ve given Claude a task where its first answer is wrong. Your job is to find it using the checklist.

  1. Ask Claude to join flights.csv with airlines.csv and count flights per airline
  2. Something is off. Verify before you believe it.
  3. When you catch it, tell Claude specifically what’s wrong and have it fix it
  4. Commit the fix with a message that says what was broken

The catch is the whole job

An agent can write the code. It cannot know if the answer is true.

That part is yours. It’s what you’re actually being graded on.
